# Veridian — Journeys System: Complete Implementation Guide
# For Cursor AI — Read This Before Building Any Journey-Related Feature

---

## 1. MENTAL MODEL — Read This First

Before touching any code, internalize this hierarchy:

```
Journey
  └── Module (many, each progresses independently)
        └── Activity (several per module, each a different type)
              └── Session (one saved record per completed attempt)
                    └── Card (only for flashcardSet activities — FSRS-tracked)
```

**The rules that govern everything:**

- A Journey is a study container for one subject or exam. It owns modules, journey-level activities, and metadata.
- A Module is one unit of study (chapter, topic, concept cluster). It progresses independently through Stage A → B → C. Stage is per-module, never per-journey.
- An Activity is a persistent, reusable learning surface. It is a real database record. It does not get recreated every session. It is the thing the student launches.
- A Session is one saved run of an activity. When a student completes a quiz, a feynman session, a flashcard review — a session record is written. Sessions accumulate over time and power history views.
- Cards are FSRS-tracked items. They belong to a flashcardSet activity. Each card holds its own FSRS scheduling state.
- Activities can be scoped to a Module or to the Journey. Journey-scoped activities (Interleaved Review, Journey Challenge) live on the Journey Detail page and do not belong to a specific module.

**Critical: activities are not content. They are containers.**
A practiceQuiz activity does not store a static list of questions. It stores configuration and stats. Fresh questions are generated by Gemini every time the student launches a session. The session record then stores the actual questions, answers, and results from that specific run.

## 2. ACTIVITY TYPES — Complete Enum

The Activity.type field must support exactly these values:

```
learningGuide       — Stage A. One per module. Generated lazily.
practiceQuiz        — Stage B. One per module. Fresh questions per session.
flashcardSet        — Stage B. Zero or more per module. Student-created, AI-assisted.
feynman             — Stage C. One per module.
freeRecall          — Stage C. One per module.
synthesis           — Stage C. One per module.
interleavedReview   — Journey-scoped. One per journey.
journeyChallenge    — Journey-scoped. One per journey.
```

**Do not use "masteryCheck" anywhere.** It was in the original schema draft and is too vague. Feynman, freeRecall, and synthesis are each their own distinct type.

---

## 3. ACTIVITY SCOPE

Activities belong to either a module or a journey. The Activity entity needs a `scope` field:

```
scope: "module" | "journey"
```

- Module-scoped: learningGuide, practiceQuiz, flashcardSet, feynman, freeRecall, synthesis
- Journey-scoped: interleavedReview, journeyChallenge

When scope is "journey", the moduleId field is null. The journeyId is always present on every activity regardless of scope.

## 4. WHAT GETS CREATED AT JOURNEY GENERATION

When a student finishes the journey creation flow and the AI processes their source material, the following is written to the database:

**Always created immediately:**
- Journey record
- Module records (one per AI-proposed module)
- Per-module knowledgeMap populated on each Module record
- One practiceQuiz activity record per module (status: ready)
- One feynman activity record per module (status: ready)
- One freeRecall activity record per module (status: ready)
- One synthesis activity record per module (status: ready)
- One interleavedReview activity record scoped to the journey (status: ready)
- One journeyChallenge activity record scoped to the journey (status: ready)

**Created lazily (not at generation time):**
- learningGuide activity — created only when the module's diagnostic result or student action confirms Stage A is needed. Created with status "notGenerated" immediately, then the guide content is populated when the student first opens it or when the system detects they will need it.

**Never created automatically:**
- flashcardSet activities — NEVER auto-created. Only created when the student explicitly creates a deck from inside a module. There is no default deck. Some modules will have zero flashcard sets forever.

## 5. LEARNING GUIDE — Detailed Behavior

**When it gets generated:**
The learningGuide activity record is created for every module at journey creation with status "notGenerated". The actual content (the guide sections) is generated by Gemini the first time the student opens the module view or the first time the diagnostic places them in Stage A.

If generation is in progress when the student navigates to it, show a loading state inline — not a full page spinner.

**Content structure (stored in Activity.content):**
```json
{
  "sections": [
    {
      "sectionId": "string",
      "title": "string",
      "explanation": "string (plain language, LaTeX where needed)",
      "workedExamples": [
        {
          "scenario": "string",
          "steps": ["string"],
          "answer": "string",
          "reasoning": "string"
        }
      ],
      "checkInQuestion": {
        "question": "string",
        "type": "multipleChoice | shortAnswer | reflection",
        "options": ["string"] or null,
        "correctAnswer": "string",
        "explanation": "string"
      },
      "externalSearchSuggestions": ["string"],
      "narrationText": "string"
    }
  ],
  "totalSections": number,
  "estimatedMinutes": number
}
```

**Behavior rules:**
- Student progresses section by section. Each section can be marked complete manually or automatically after reading + check-in.
- Student can skip any section. Veridian shows a soft note: "Stage A was recommended for this module based on your diagnostic." No blocking.
- Student can return to any section at any time after completion. The guide is never locked or hidden.
- After completing the guide, the Stage A summary screen appears with a CTA to begin Stage B. This does NOT auto-advance. The student confirms.
- Completion state is stored as `completedSectionIds` in the user's progress for this activity (store on Activity or in Session depending on implementation).
- Audio narration: use Web Speech Synthesis API on the narrationText field. Play/pause per section. No external service.

## 6. PRACTICE QUIZ — Detailed Behavior

**One practice quiz activity per module. Always exactly one.**

**Activity.content stores quiz configuration and preferences:**
```json
{
  "lastConfig": {
    "questionCount": 25,
    "focusPreset": "weakSpots",
    "timedMode": false,
    "timeLimitMinutes": null
  },
  "recommendedFocus": "weakSpots | newMaterial | fullReview",
  "generationConstraints": {
    "excludedConceptIds": []
  }
}
```

**Launch flow:**
1. Student taps "Start Practice Quiz" from the module activity list.
2. Setup screen appears prefilled with lastConfig (or defaults on first launch).
3. Student can edit: question count (10/25/50/custom), focus preset (Full Review / Weak Spots / New Material), timed mode toggle.
4. Student hits "Start" — Gemini generates a fresh question set based on: module knowledge map + focusPreset + weak concept data from recent sessions + generationConstraints.
5. Questions are served one at a time. Each question shows type, stem, options (if MC), LaTeX rendered.
6. After answering: immediate feedback shown (correct/incorrect + explanation).
7. If student misses 3+ questions on the same concept: soft intervention inline — "You've missed a few questions on [concept]. Want a quick refresher?" — student chooses "Show me" (inline recap) or "Keep going".
8. On completion: session summary screen (see Session section below).
9. Session is written to database on completion. The Activity.content.lastConfig is updated with the settings used.

**Question generation rules (pass these in the Gemini prompt):**
- Never repeat a question already seen in the last 3 sessions for this module.
- Prioritize: (1) concepts due for reinforcement, (2) concepts with recent misses, (3) concepts not yet practiced enough, (4) general balance.
- Mix question types: multiple choice, short answer, true/false, multi-select, scenario-based, calculation-heavy.
- All math/science notation must use LaTeX delimiters.
- Return structured JSON array. Validate with Zod before use.

## 7. FLASHCARD SETS — Detailed Behavior

**Zero or more per module. Never auto-created. Student-initiated only.**

**How a student creates a flashcard set:**
1. From the module activity list (Stage B section), there is an "+ Add Flashcard Deck" button.
2. Student enters a deck name (e.g., "Formulas", "Vocab", "Reaction Types").
3. Student chooses: "Generate from my material" (AI populates cards from knowledge map) or "Start blank" (manual entry).
4. If AI generation: Gemini reads the module knowledge map and source material and generates card pairs. Student reviews and can edit/delete before saving.
5. Deck is saved as a flashcardSet activity record. Cards are written as Card records linked to this activity.

**Card structure (already in Card.jsonc — confirm these fields are all present):**
- cardId, activityId, journeyId, userEmail
- front (string, LaTeX supported)
- back (string, LaTeX supported)
- conceptTag (optional link to knowledge map concept)
- fsrsState: { stability, difficulty, due, reps, lapses, state, lastReview }
- suspended (boolean — excluded from scheduling if true)

**Review flow:**
1. Student opens a flashcard set. Options: "Review Due Cards" (FSRS-scheduled) or "Browse All Cards".
2. Review Due Cards: shows only cards where fsrsState.due <= now and suspended = false.
3. Card shown front-side. Student attempts recall. Taps to reveal back.
4. Student rates: Again / Hard / Good / Easy.
5. ts-fsrs calculates new scheduling. Card FSRS state is updated immediately.
6. After all due cards reviewed: session written to database.

**FSRS scheduling:**
- Use ts-fsrs library. All scheduling logic runs client-side in src/utils/fsrs/.
- Cards with due <= today's date show up in Due Today counts on Home.
- Card health for UI indicators: Green = stability > 21 days. Yellow = stability 7–21 days. Red = stability < 7 days or lapses >= 2.

**Multiple decks per module:**
Each deck is a separate activity record. Each has its own session history. UI shows them as separate entries in the Stage B list with their individual name, card count, due count, and health.

## 8. FEYNMAN MODE — Detailed Behavior

**One feynman activity per module.**

**Launch flow:**
1. Student opens Feynman Mode from Stage C activity list.
2. Concept selector shown: list of concepts from the module knowledge map. Student picks which concept to explain (or Veridian recommends the weakest one).
3. Input area: text area + microphone button (Web Speech API).
4. Voice input: microphone activates SpeechRecognition, transcript streams into text area in real time. Student can edit before submitting.
5. Student submits explanation.
6. Gemini evaluates: identifies missing pieces, vagueness, misconceptions, weakest specific point.
7. AI response shown: "Good — you covered X and Y. Your explanation of Z is where it breaks down." Plus a targeted follow-up question.
8. Student answers follow-up if prompted.
9. Session summary shown. Session written to database.

**Session data to save:**
- conceptPrompt (which concept was chosen)
- rawStudentResponse (typed or transcribed text)
- wasVoiceInput (boolean)
- aiFeedback (AI's full evaluation text)
- missingConcepts (array)
- misconceptionsDetected (array)
- weakestPoint (string — the specific thing AI flagged)
- followUpQuestion (string)
- followUpResponse (string or null if skipped)
- overallConfidenceRating ("strong" | "partial" | "weak")

## 9. FREE RECALL — Detailed Behavior

**One freeRecall activity per module.**

**Launch flow:**
1. Student opens Free Recall from Stage C activity list.
2. Topic prompt shown: either a module name, a subtopic, or a specific concept cluster. Veridian selects based on what was studied least recently.
3. Blank text area with microphone option.
4. Hint button available. Tiers:
   - Hint 1: reveals a section name or subtopic label
   - Hint 2: reveals a key term
   - Hint 3: reveals a one-sentence conceptual nudge
   - Hints do not invalidate the session. Number of hints used is recorded.
5. Student submits when done.
6. Gemini compares recall against module knowledge map: concepts covered, concepts missed, inaccuracies.
7. Session summary shown with coverage estimate and recommended next concept.
8. Session written to database.

**Session data to save:**
- recallPrompt (topic/concept shown)
- rawStudentResponse
- wasVoiceInput (boolean)
- hintsUsed (number)
- hintTiersRevealed (array of which hints were shown)
- coveragePercent (0–100)
- conceptsCovered (array)
- conceptsMissed (array)
- incorrectIdeas (array)
- aiGradingSummary (string)
- nextConceptRecommendation (string)

## 10. SYNTHESIS QUESTIONS — Detailed Behavior

**One synthesis activity per module.**

**Launch flow:**
1. Student opens Synthesis from Stage C activity list.
2. Gemini generates questions that combine concepts across multiple modules in the same journey. Requires at least two modules with some Stage B activity.
3. Questions require comparing/applying/choosing between concepts across modules.
4. Student answers each question.
5. Session summary emphasizes which module combinations held up and where integration broke down.
6. Session written to database.

**Session data to save:**
- questions (full array with stems, options, explanations)
- answers (student selections)
- correctness (per question)
- modulesInvolved (array of moduleIds pulled into this session)
- crossModuleConfusionNotes (string — AI description of where integration failed)
- integrationReadinessSignal ("strong" | "partial" | "weak")

## 11. INTERLEAVED REVIEW — Detailed Behavior

**One interleavedReview activity, scoped to the Journey (moduleId = null).**

**Available when:** At least two modules in the Journey are in Stage B.

**Launch flow:**
1. Student opens Interleaved Review from the Journey Detail page (bottom action row).
2. Setup: select which modules to include (checkboxes, all pre-checked), choose question count, timed or untimed.
3. Veridian recommends modules that are both important and unstable.
4. Gemini generates a mixed question set pulling from all selected modules' knowledge maps.
5. Questions are served in random order, not grouped by module.
6. On completion: session summary showing which modules held up under mixing and which collapsed.
7. Session written to database with moduleId = null, journeyId = journey.

**Session data to save:**
- selectedModuleIds (array)
- questions (full array)
- answers, correctness, explanations
- perModuleAccuracy ({ [moduleId]: percent })
- weakModulesUnderMixing (array)
- crossModuleConfusionNotes (string)

## 12. JOURNEY CHALLENGE — Detailed Behavior

**One journeyChallenge activity, scoped to the Journey (moduleId = null).**

**Available when:** At least two modules have meaningful Stage B activity.

**Launch flow:**
1. Student opens Journey Challenge from Journey Detail page.
2. Setup: short/medium/long, timed or untimed, weighting (balanced vs. bias toward weak areas).
3. Default is balanced across all modules.
4. On completion: module-by-module breakdown, overall readiness estimate, recommended next action.
5. Session written to database.

**Session data to save:**
- Same structure as interleavedReview but with an added overallReadinessSignal ("examReady" | "nearlyReady" | "notReady") and a recommendedStudyPlan (string).

## 13. SESSION ENTITY — Updated Required Fields

The current Session.jsonc schema is too thin. Here is the complete required shape:

```
sessionId           string    required
journeyId           string    required
moduleId            string | null   null for journey-scoped activities
activityId          string    required
userEmail           string    required
activityType        string    mirrors Activity.type enum
startedAt           number    unix ms
endedAt             number    unix ms
durationSec         number
status              string    "completed" | "abandoned"
score               number | null   0-100 where applicable
outcomeSummary      object    { accuracy?, itemsCompleted?, weakConcepts[], strongConcepts[], nextAction? }
sessionData         object    type-specific rich payload (see sections 6-12 above for each type)
```

`sessionData` is where all the rich, type-specific content lives. `outcomeSummary` is a lightweight denormalized version used for quick display in history lists without parsing the full sessionData.

## 14. ACTIVITY ENTITY — Updated Required Fields

The current Activity.jsonc schema needs these additions:

```
activityId          string    required
moduleId            string | null   null when scope = "journey"
journeyId           string    required
userEmail           string    required
type                string    full enum from Section 2
scope               string    "module" | "journey"
status              string    "notGenerated" | "generating" | "ready" | "failed"
title               string    display name for UI (e.g. "Practice Quiz", "Formulas Deck")
description         string    one-liner for UI
content             object    type-specific configuration and stored content (see each section)
itemCount           number    cards for flashcardSet, sections for learningGuide, null for others
stats               object    { totalSessions, avgAccuracy, lastScore, lastCompletedAt, dueCount? }
lastSessionAt       number    denormalized for fast display
createdAt           number
updatedAt           number
```

The `stats` field is denormalized and must be updated after every session completion. Do not compute stats by aggregating sessions at read time — too expensive. Update stats synchronously when writing a session.

## 15. UPDATED SCHEMA DECISIONS

### Activity.type enum (final)
```
learningGuide | practiceQuiz | flashcardSet | feynman | freeRecall | synthesis | interleavedReview | journeyChallenge
```

### Module.masteryScore derivation
Mastery is NOT only from FSRS. It is a weighted composite of:
- practiceQuiz: average accuracy across recent sessions (last 3)
- flashcard health: average FSRS stability across non-suspended cards (only if decks exist)
- feynman: overallConfidenceRating from most recent session (strong=100, partial=60, weak=20)
- freeRecall: coveragePercent from most recent session

If a module has no flashcard decks, that component is excluded from the composite. The masteryScore field stores 0-100.

### Module.stage derivation
Stage is set by diagnostic result at Journey creation. It can also be manually updated if the student explicitly moves forward or backward. It is not auto-calculated — it is a deliberate placement.

## 16. JOURNEY DETAIL PAGE — UI Specification

**Route:** /journeys/:id

**Page Header:**
- Journey title (editable inline via pencil icon)
- Subject label
- Exam date shown as: "June 20 — 13 days remaining" / "No deadline set"
- Overall mastery: percentage derived from average of all Module.masteryScore values
- Readiness status: "On track" | "Needs attention" | "Behind — focus required" (derived from deadline proximity + mastery)
- Gear icon for Journey Settings

**Recommended Study Plan panel (directly below header):**
This is the deadline-aware scheduling layer. It only lives on this page.
- Today's recommended sessions: 1-3 specific activity recommendations. Each shows module name + activity type + why it's recommended (e.g. "3 flashcards due", "weak concepts detected", "not practiced in 4 days")
- This week's priorities: which modules need work before deadline
- If no deadline: panel shows "Suggested Next" — the single most productive next activity across all modules
- Plan recalculates after every session completion

**Scheduling logic (implement in src/utils/studyPlanner.js):**
Inputs: Journey.examDate, all Module.masteryScore values, all Module.stage values, all recent Session records, all Card FSRS due dates
Output: ordered list of recommended sessions for today and this week

Priority order:
1. Cards due today (FSRS)
2. Modules with mastery < 40% and deadline within 14 days
3. Modules not studied in 5+ days with an upcoming deadline
4. Lowest mastery modules in Stage B
5. Modules ready to advance to next stage

**Module list (below study plan):**
Each module shown as an expandable row/card:

Collapsed state shows:
- Module name
- One-line description
- Stage badge (A / B / C) in a colored pill
- Mastery percent
- Last studied date (relative: "2 days ago")
- Health dot (green / yellow / red)
- "Start Session" button — launches the recommended next activity for this module without navigating away

Expanded state (clicking module card, not the Start button) shows:
- Same header info
- Per-activity quick stats: last quiz score, flashcard due count, last feynman date, etc.
- "Go to Module" button → navigates to Module Detail page

**Journey-level actions (below module list):**
- "Interleaved Review" button (grayed if < 2 modules in Stage B)
- "Journey Challenge" button (grayed if < 2 modules have Stage B activity)
- "Cram Mode" button (always available)
- "Publish to Community" button (only if Journey meets minimum requirements: 3+ modules, 1+ complete activity per module, 1+ tag)
- "Journey Insights" — collapsible panel: session history timeline, accuracy trends per module, FSRS health summary

## 17. MODULE DETAIL PAGE — UI Specification

**Route:** /journeys/:id/modules/:moduleId

**Breadcrumb:** Journey title → Module name

**Page Header:**
- Module name
- AI-generated one-line summary of what this module covers
- Stage badge
- Mastery percent
- Health dot
- Last studied date

**Stage sections (three visible sections, always):**

All three stages are ALWAYS visible. None are locked or hidden. The current recommended stage is visually highlighted (brighter, expanded by default). Stages not yet recommended are shown dimmed but fully accessible. First time a student accesses a non-recommended stage, a soft inline note appears: "Veridian recommends completing Stage A first — but you can start here." Note disappears after one session in that stage.

**Stage A — Learn:**
Contains: learningGuide activity
- Shows: estimated read time, section count, narration available (boolean), completion status (Not Started / In Progress / Completed), last opened date
- If status = notGenerated: shows "Generating your Learning Guide..." with spinner, or "Generate Learning Guide" button if not yet triggered
- If status = ready: shows "Continue" or "Start" button
- After first completion: "Completed ✓" label remains, but "Review" button still available — guide is never locked

**Stage B — Practice:**
Contains: practiceQuiz, zero or more flashcardSet activities

Practice Quiz entry shows:
- "Practice Quiz" title
- "Recommended: Weak Spots" (or whatever focus is recommended based on recent performance)
- Sessions completed count
- Most recent accuracy
- "Start" button → opens setup screen (prefilled with last config or defaults)

Each flashcardSet entry shows:
- Deck name (e.g. "Formulas")
- Card count total
- Cards due today (from FSRS)
- Health dot
- "Review Due Cards" primary button
- "Browse All Cards" secondary button
- Edit (pencil) icon → opens deck editor where student can edit/add/delete cards
- Delete (trash) icon with confirmation

"+ Add Flashcard Deck" button always visible at bottom of Stage B section.

**Stage C — Mastery:**
Contains: feynman, freeRecall, synthesis activities

Each entry shows:
- Activity name
- One-line purpose description
- Last completed date (or "Not attempted yet")
- Readiness note: "Module mastery looks stable — good time to try this" or "Consider more Stage B practice first"
- "Start" button

Synthesis entry also shows: "Requires 2+ modules with Stage B activity" if condition not yet met.

**Session History panel (collapsible, collapsed by default):**
Shows chronological log of all sessions for this module. Each entry:
- Date and relative time
- Activity type (icon + label)
- Duration
- Outcome summary (e.g. "18 questions — 74% accuracy", "12 cards — 3 Again", "Feynman — Partial confidence")
- Expand arrow

Expanding a Practice Quiz, Feynman, Free Recall, or Synthesis session shows full sessionData:
- For quiz: every question, student answer, correct/incorrect indicator, correct answer, explanation
- For feynman: student response, AI feedback, weakest point, follow-up Q&A
- For freeRecall: student response, coverage %, missed concepts, incorrect ideas
- For flashcard: cards reviewed, ratings given, FSRS changes

History entries are read-only records. They cannot be re-launched. Each session is unique and sealed after completion.

**Concept Map panel (collapsible, at bottom of page):**
Shows the module knowledge map as a structured list:
- Each concept with its current mastery state: Strong (green) / Moderate (yellow) / Weak (red) / Not Practiced (gray)
- Clicking a concept shows: relevant flashcard cards linked to it (via conceptTag), and any sessions where this concept appeared as weak

**Opens as a full page (not modal) from "+ New Journey" button.**

**Step 1 — Name and Deadline:**
- Journey title field (required, placeholder: "e.g. AP Chemistry, MCAT Biochemistry")

- Progress indicator

## 21. AI CALL LOCATIONS

All AI calls live in src/api/ai/. Never call Gemini from a component or hook.

Functions needed:
- `extractKnowledgeMap(sourceContent, fileUrls[])` → structured JSON knowledge map
- `scaffoldJourney(knowledgeMap)` → module names, descriptions, concept assignments
- `generateLearningGuide(moduleKnowledgeMap, sourceContent)` → sectioned guide content
- `generatePracticeQuestions(moduleKnowledgeMap, config, recentSessionData)` → question array
- `generateFlashcards(moduleKnowledgeMap, deckName)` → card pairs array
- `gradeFeymanExplanation(concept, studentResponse, moduleKnowledgeMap)` → feedback object
- `gradeFreeRecall(prompt, studentResponse, moduleKnowledgeMap)` → coverage analysis object
- `generateSynthesisQuestions(moduleKnowledgeMapArray)` → question array
- `generateInterleavedQuestions(moduleKnowledgeMapArray, config)` → question array
- `analyzeSessionPatterns(sessionArray)` → insights string

Every AI function must:
- Request JSON output via responseMimeType: "application/json"
- Validate response with Zod schema before use
- Have a fallback if validation fails (retry once, then surface graceful error)
- Be wrapped in try/catch with meaningful error states shown in UI

## 22. FSRS IMPLEMENTATION

## 23. MASTERY SCORE CALCULATION

Lives in src/utils/mastery.js.

`calculateModuleMastery(module, activities, recentSessions, cards)` → number 0-100

Components:
- Quiz component (weight ~40%): average accuracy from last 3 practiceQuiz sessions for this module. If no sessions, contribute 0.
- Flashcard component (weight ~30%): average FSRS stability normalized to 0-100 across all non-suspended cards for this module. Exclude entirely if no flashcard decks exist for this module (redistribute weight to other components).
- Feynman component (weight ~15%): map most recent feynman session's overallConfidenceRating: strong=100, partial=60, weak=20. If no sessions, contribute 0.
- Free Recall component (weight ~15%): most recent freeRecall session's coveragePercent. If no sessions, contribute 0.

If a component has no data, redistribute its weight proportionally across components that do have data. A module with no activity at all has masteryScore = 0.

Recalculate and persist Module.masteryScore after every relevant session completion.

## 25. JOURNEY CARD NOTE LOGIC (for Journeys List page)

Derive the status note on each Journey card in this priority order:

```javascript
function getJourneyStatusNote(journey, modules, cards) {
  const daysRemaining = journey.examDate
    ? Math.ceil((journey.examDate - Date.now()) / 86400000)
    : null;

  const overdueCards = cards.filter(c =>
    c.journeyId === journey.journeyId &&
    c.fsrsState.due <= Date.now() &&
    !c.suspended
  );

  const daysSinceStudied = journey.lastStudiedAt
    ? Math.floor((Date.now() - journey.lastStudiedAt) / 86400000)
    : null;

  const avgMastery = average(modules.map(m => m.masteryScore));
  const modulesNeedingAttention = modules.filter(m => m.masteryScore < 40).length;

  if (daysRemaining !== null && daysRemaining <= 7) {
    return { text: `Exam in ${daysRemaining} day${daysRemaining !== 1 ? "s" : ""} — review recommended`, color: "red" };
  }
  if (overdueCards.length > 0) {
    const moduleCount = new Set(overdueCards.map(c => c.moduleId)).size; // rough grouping
    return { text: `${moduleCount} module${moduleCount !== 1 ? "s" : ""} need card review`, color: "yellow" };
  }
  if (daysSinceStudied !== null && daysSinceStudied >= 5) {
    return { text: `Last studied ${daysSinceStudied} days ago`, color: "muted" };
  }
  if (avgMastery >= 70 && modulesNeedingAttention === 0) {
    return { text: "On track", color: "green" };
  }
  return { text: "Keep studying", color: "muted" };
}
```
